# 将 PI0/PI0.5 的动作生成方法改为 Diffusion Policy 的思路

本文面向当前 `C:\QClaw\FASTER_hy` 仓库。当前 `Pi0` / `Pi0Faster` 已经不是自回归 token action，而是连续动作上的 flow matching：训练时给真实 action 加噪声，模型预测速度场 `v_t`；推理时从高斯噪声迭代到完整 action chunk。核心代码在：

- `src/openpi/models/pi0.py`
  - `Pi0.compute_loss()`：构造 `x_t = t * noise + (1 - t) * actions`，监督 `u_t = noise - actions`
  - `Pi0.sample_actions()`：从 noise 开始，用预测速度迭代得到 action chunk
- `src/openpi/models/pi0_faster.py`
  - 在 PI0 flow matching 基础上加入 HAS、prefix action、streaming
- `src/openpi/training/config.py`
  - `pi05_libero`、`pi05_libero_low_mem_finetune`、`pi05_faster_libero` 等训练配置

所以“换成 diffusion policy”可以有多种层级：最小改动是把 flow matching 目标换成 DDPM/DDIM 的噪声预测；中等改动是保留 PaliGemma 作为视觉语言条件，只把 action denoiser 换成 Diffusion Policy 风格的 temporal network；最大改动是引入独立的 Torch Diffusion Policy 模型，让 openpi 只负责服务和数据格式。

## 方案一：最小改动，把 flow matching 换成 DDPM/DDIM 噪声预测

### 核心想法

保留当前 `Pi0` 的绝大部分结构：

```text
image + prompt + state -> PaliGemma prefix
noisy_actions + diffusion timestep -> action expert suffix
action_out_proj -> action prediction
```

但改变训练目标：

```text
当前 flow matching:
  x_t = t * noise + (1 - t) * action
  target = noise - action
  model predicts velocity v_t

DDPM epsilon prediction:
  x_t = sqrt(alpha_bar_t) * action + sqrt(1 - alpha_bar_t) * noise
  target = noise
  model predicts epsilon
```

推理时也从 Euler flow 更新换成 DDIM/DDPM 更新。

### 需要改的代码

建议新建文件，而不是直接改 `pi0.py`：

```text
src/openpi/models/pi0_diffusion.py
src/openpi/models/pi0_diffusion_config.py  或扩展 pi0_config.py
```

可以从 `Pi0` 复制出 `Pi0Diffusion`：

- 复用 `embed_prefix()`
- 复用 `embed_suffix()`
- 复用 `PaliGemma.llm`
- 改写 `compute_loss()`
- 改写 `sample_actions()`

新增 diffusion schedule helper：

```python
def cosine_beta_schedule(num_train_timesteps: int):
    ...

def q_sample(x0, t_idx, noise, alpha_bar):
    return sqrt_alpha_bar[t_idx] * x0 + sqrt_one_minus_alpha_bar[t_idx] * noise
```

训练时：

```text
t_idx ~ Uniform(0, T-1)
noise ~ Normal(0, 1)
x_t = q_sample(actions, t_idx, noise)
eps_pred = model(observation, x_t, t_idx)
loss = mse(eps_pred, noise)
```

推理时：

```text
x_T ~ Normal(0, 1)
for t = T-1 ... 0:
  eps_pred = model(observation, x_t, t)
  x_{t-1} = DDIM/DDPM update(x_t, eps_pred, t)
return x_0
```

### 优点

- 改动最小，仍走现有 `Policy.infer()`、数据 transforms、checkpoint、LoRA、serve 逻辑。
- 可以继续用 `pi05_base` 或 `pi05_libero` 初始化大部分权重。
- 仍然保留 VLA 的视觉语言能力。

### 风险

- `pi05_base` 是按 flow matching 训练的，不是按 DDPM 训练的。直接改目标后，原 action expert 的预训练优势可能打折。
- 推理步数可能需要更多，例如 20、50、100 步，延迟会增加。
- 当前 FASTER 的 HAS/streaming 是围绕 flow matching 速度场设计的，不能原样套到 DDPM。

### 推荐用途

这是最适合作为第一版实验的方案。目标是验证“同一套 VLA 条件编码 + diffusion objective”能不能在 LIBERO 上跑通。

## 方案二：保留 PaliGemma 条件编码，替换为 Diffusion Policy 风格 action head

### 核心想法

把 `Pi0` 拆成两部分：

```text
视觉语言条件编码器:
  image + prompt + state -> condition embedding

动作扩散模型:
  noisy action sequence + diffusion timestep + condition embedding
  -> predicted noise / predicted action
```

这里的动作扩散模型不再使用 Gemma action expert，而是改成 Diffusion Policy 常见结构：

- 1D Conditional U-Net
- Temporal Transformer denoiser
- MLP-Mixer style action denoiser

动作 shape 仍保持 openpi 约定：

```text
[batch, action_horizon, action_dim]
```

LIBERO 中实际动作是 7 维，进入模型前会 pad 到 32 维；输出后 `LiberoOutputs()` 只取前 7 维。

### 需要改的代码

新增模型：

```text
src/openpi/models/pi0_dp_head.py
```

结构上可以是：

```python
class Pi0DiffusionPolicy(BaseModel):
    def __init__(...):
        self.PaliGemma = ...
        self.condition_proj = ...
        self.action_denoiser = ConditionalTemporalUNet(...)

    def encode_condition(obs):
        prefix_tokens = self.embed_prefix(obs)
        condition = pool(prefix_tokens)  # mean pool / CLS-like token / attention pool
        return condition

    def compute_loss(...):
        condition = encode_condition(observation)
        x_t, noise, t = diffusion_forward(actions)
        eps_pred = action_denoiser(x_t, t, condition)
        return mse(eps_pred, noise)

    def sample_actions(...):
        condition = encode_condition(observation)
        x = normal(...)
        for t in reverse_schedule:
            eps = action_denoiser(x, t, condition)
            x = sampler_step(x, eps, t)
        return x
```

### 条件向量怎么取

有三种选择：

1. mean pool 所有有效 prefix tokens
2. 取 language 最后一个有效 token
3. 加一个 learned query，对 prefix tokens 做 cross-attention pooling

建议第一版用 mean pool，最稳、最少代码。

### 优点

- 更接近传统 Diffusion Policy。
- 动作生成模块更轻，可能比完整 Gemma action expert 省显存。
- 可以冻结视觉语言大模型，只训练 DP head，训练更像 low-memory fine-tune。

### 风险

- 不能直接复用 `pi05_libero` 的 action expert 能力，因为 action head 架构变了。
- 需要从头训练 DP head，10 step、几百 step 仍然不会有成功率。
- 如果只用 pooled condition，可能丢失 token-level 细节，精度低于原 PI0。

### 推荐用途

当目标是“真正研究 Diffusion Policy action head 是否优于 PI0 flow matching head”时，选这个方案。

## 方案三：混合方案，用 Gemma action expert 做 diffusion denoiser

### 核心想法

不完全丢掉 PI0 的 action expert，而是让它预测 DDPM 的 `epsilon`、`x0` 或 `v`。

常见三种 parameterization：

```text
epsilon prediction:
  model predicts noise

x0 prediction:
  model predicts clean action

v prediction:
  model predicts diffusion v-parameterization
```

当前 PI0 的 flow matching 输出已经是速度场，最接近 diffusion 的 `v-prediction` 思路。因此可以做一个比较温和的改法：

```text
保留 embed_suffix(noisy_actions, timestep)
保留 PaliGemma action expert
保留 action_out_proj
只替换 noise schedule 和 loss target
```

### 具体落实

当前按“最小结构改动”落地为一条独立模型路线：

```text
src/openpi/models/pi0.py
-> 复制为 src/openpi/models/pi0_diffusion.py
-> 类名改为 Pi0Diffusion
-> 保留 embed_prefix / embed_suffix / PaliGemma action expert / action_out_proj
-> 只替换 compute_loss 和 sample_actions 的动作生成目标
```

配置侧新增：

```text
src/openpi/models/pi0_config.py
  Pi0DiffusionConfig(Pi0Config)

src/openpi/training/config.py
  pi05_libero_diffusion
  pi05_libero_diffusion_lora
```

`Pi0DiffusionConfig` 当前字段：

```python
num_diffusion_train_timesteps: int = 100
diffusion_prediction_type: Literal["epsilon", "sample", "v"] = "epsilon"
diffusion_schedule: Literal["cosine", "linear"] = "cosine"
diffusion_min_alpha_bar: float = 1e-4
diffusion_clip_sample: bool = False
```

`diffusion_prediction_type="epsilon"` 是第一版默认值，主要是为了和熟悉的 PADP 代码对齐：

```text
C:\QClaw\PADP\diffusion_policy\policy\dp_diffusion_unet_image_policy.py
  training:
    noise = torch.randn(...)
    timesteps = torch.randint(...)
    noisy_trajectory = noise_scheduler.add_noise(trajectory, noise, timesteps)
    pred = model(noisy_trajectory, timesteps, ...)
    target = noise

  inference:
    trajectory = torch.randn(...)
    scheduler.set_timesteps(num_inference_steps)
    for t in scheduler.timesteps:
      model_output = model(trajectory, t, ...)
      trajectory = scheduler.step(model_output, t, trajectory).prev_sample
```

在 JAX 版 `Pi0Diffusion` 中，对应关系是：

```text
training:
  time = (timestep + 1) / num_diffusion_train_timesteps
  alpha_bar = alpha_bar_fn(time)
  x_t = sqrt(alpha_bar) * actions + sqrt(1 - alpha_bar) * noise
  target = noise / actions / v，取决于 diffusion_prediction_type
  pred = Gemma action expert(observation, x_t, time)
  loss = mse(pred, target)

inference:
  x_t 从 Gaussian noise 开始
  每步用 Gemma action expert 预测 model_output
  用 deterministic DDIM 公式从 time 更新到 next_time
  最终返回归一化空间的连续 action chunk
```

这条路线仍然返回 `[batch, action_horizon, action_dim]`，因此 `Policy.infer()`、`serve_policy.py` 和 output transforms 不需要为 diffusion 单独改。

### 优点

- 比方案二更容易复用 PI0 权重。
- 仍能使用 LoRA 微调。
- 可以和当前 `pi05_libero_low_mem_finetune` 类似，做低显存实验。

### 风险

- 代码上容易和当前 flow matching 混淆，需要清晰命名。
- 如果 sampler 实现不严谨，训练 loss 会下降但动作质量很差。
- FASTER 的 streaming 仍不能直接复用。

### 推荐用途

这是最适合对比“flow matching vs diffusion objective”的路线。

## 方案四：引入独立 Torch Diffusion Policy，让 openpi 只做服务/数据适配

### 核心想法

不在 JAX/NNX 里改 PI0，而是引入已有 Torch Diffusion Policy 实现，例如：

```text
observation image/state/prompt
-> openpi transforms / adapter
-> Torch DP policy
-> action chunk
-> LiberoOutputs
```

也可以参考当前工作区里的其他 DP/PADP 代码，把它作为单独 policy server。

### 两种接法

#### 接法 A：单独 server

保持 openpi 不动，另开一个 DP server 实现同样的 websocket 协议：

```text
examples/libero/main.py -> websocket client -> DP server -> actions
```

优点是改动少，最快能跑 LIBERO。

#### 接法 B：接入 openpi Policy

新增一个 `BaseModel.load_pytorch()` 或新的 policy wrapper，让 `scripts/serve_policy.py` 能加载 Torch DP checkpoint。

当前 `policy_config.py` 已经有检测 `model.safetensors` 的 PyTorch 分支，但它服务的是 openpi 的 PyTorch PI0 模型，不是通用 DP。要接 DP，建议新建：

```text
src/openpi/policies/diffusion_policy_wrapper.py
scripts/serve_diffusion_policy.py
```

### 优点

- 可以复用成熟 DP 代码和调参经验。
- 不受 JAX/NNX/Orbax 约束。
- 适合快速验证“LIBERO 上 DP baseline 能不能跑”。

### 风险

- 和 openpi checkpoint、assets、norm stats 的契约容易不一致。
- prompt/VLM 能力可能变弱，除非额外接 CLIP/T5/VLM encoder。
- 多环境依赖会复杂，尤其 Windows、CUDA、MuJoCo、Torch 版本。

### 推荐用途

如果目标是工程上尽快得到一个 diffusion policy baseline，而不是保留 PI0/VLA 结构，选这个方案。

## 方案五：把 Diffusion Policy 与 FASTER/streaming 结合

### 核心想法

FASTER 的核心是降低 TTFA：近端 action 更早完成，远端 action 继续 denoise。当前 `Pi0Faster` 通过 HAS 给不同 horizon 位置分配不同 timestep。

如果换成 DDPM/DDIM，也可以设计类似机制：

```text
普通 diffusion:
  所有 horizon 位置共享同一 denoising timestep

horizon-aware diffusion:
  每个 horizon 位置有自己的 diffusion timestep
  近端 action 更快走到 t=0
  远端 action 保留更高噪声继续迭代
```

### 可行做法

1. 先实现非 streaming 的 diffusion policy。
2. 把 timestep 从 `[batch]` 扩展到 `[batch, action_horizon]`。
3. 参考 `Pi0Faster.compute_HAS()`，为每个 horizon 构造不同 diffusion index。
4. 推理时每一步检查哪些 action 已经到达 `t=0`，提前输出。

### 主要难点

- DDPM/DDIM 的更新公式通常默认一个全局 timestep。扩展到每个 horizon 独立 timestep 后，要非常小心 `alpha_bar_t` 的 gather 和广播。
- 如果 action token 之间有自回归/temporal attention，早发 action 与后续 action 的一致性可能变差。
- 需要重新设计 prefix action mask 和 loss mask。

### 推荐用途

不要作为第一步。建议先在普通 PI0/DP 上验证成功率，再研究 streaming。

### 第一步：做方案一或方案三的最小版本

目标：复用当前 `Pi0` 结构，只改 loss 和 sampler。

建议新增：

## 最推荐的第一版设计

我建议第一版选“方案三：Gemma action expert + diffusion objective”，原因是：

1. 改动比完整 DP head 小。
2. 可以最大限度复用 `pi05_base` / `pi05_libero` 权重。
3. 训练、评估、serve、LoRA 都能沿用当前 openpi 体系。
4. 和当前 flow matching 的差异集中在 loss/sampler，实验结论更清晰。

第一版不要做：

- 不要同时上 FASTER streaming。
- 不要先换掉整个 PaliGemma action expert。
- 不要只跑 10 step 就判断成功率。
- 不要把 `pi05_libero` 的 config 和 LoRA diffusion config 混用。

## 预期实验结果解读

如果 10 step 成功率是 0：

```text
正常，只说明没有训练充分。
```

如果 3000 step loss 下降但成功率仍为 0：

```text
优先检查：
1. 反归一化是否正确
2. action 前 7 维是否合理
3. diffusion sampler 是否数值稳定
4. x0 / epsilon / v 参数化是否和 update 公式匹配
5. policy.config 和 checkpoint 是否匹配
```

如果 simple_client 输出 action 数值爆炸：

```text
优先检查：
1. alpha_bar schedule
2. DDIM update 公式
3. 是否忘了 clip / clamp x0
4. 是否把 timestep index 和连续 timestep 混用
```

如果显存爆炸：

```text
优先使用：
1. LoRA
2. batch_size=4 或 8
3. ema_decay=None
4. 冻结 PaliGemma，只训练 action denoiser
5. 减少 num_diffusion_infer_steps 不影响训练显存，但影响推理速度
```
